EXPERIMENT CONTEXT
"Tools or Thinking? Disentangling Search and Planning for LLM Agents over Data Lakes"

=============================================================================
EXPERIMENT GRID
=============================================================================

3 conditions × 3 models = 9 runs
Dataset: LakeQA-mini (135 tasks)
Budget: 30 tool calls per task (excluding plan, skills, summarize_context, submit_answer)
Context limit: 200k tokens

CONDITIONS:
  Baseline  — BM25 search + no context tools      (search-basic, planning-basic)
  Cond A    — Hybrid search + no context tools     (search-rich,  planning-basic)
  Cond B    — BM25 search + full context toolkit   (search-basic, planning-rich)

MODELS:
  Claude Haiku 4.5
  Claude Sonnet 4.5
  Llama-3.3-70B-Instruct

GRID:

=============================================================================
WHAT EACH CONDITION HAS
=============================================================================

BASELINE
  Search:  search_value (BM25), search_schema (BM25, first-50-row index), search_prefix
  Context: system prompt with tool descriptions only
  Purpose: control — same search tools as Cond B, no context enhancements

CONDITION A  (Search-Rich)
  Search:  search_value (hybrid RRF: BM25 + vector ANN),
           search_schema (hybrid RRF over first-50-row schema index),
           search_reranked (hybrid + cross-encoder reranker: BAAI/bge-reranker-v2-m3),
           search_prefix
  Context: system prompt with tool descriptions only
  Purpose: isolates effect of better search tools

CONDITION B  (Planning-Rich)
  Search:  search_value (BM25), search_schema (BM25), search_prefix
  Context: system prompt + plan tool + skills plugin (3 skills) +
           CategoryStagnationHandler loop nudge + summarize_context tool
  Purpose: isolates effect of richer planning/context management

SHARED DATA TOOLS (all conditions):
  list_files, peek_file, peek_files, read_file, grep_file,
  query_file, download, execute_code, get_sandbox_info, cleanup_sandbox

=============================================================================
KEY COMPARISONS
=============================================================================

Baseline vs Cond A  →  effect of search quality (same context, better search)
Baseline vs Cond B  →  effect of planning tools (same search, better context)
Cond A   vs Cond B  →  search-rich vs planning-rich (main research question)

Within each comparison, model axis shows whether effect is model-dependent
(hypothesis: planning helps smaller models more; search helps all models).

=============================================================================
SEARCH INDEX SETUP
=============================================================================

Hybrid index (Cond A):
  - Chunking: 800-word chunks for all files; 10k rows per table
  - FTS: BM25 with en_stem tokenizer
  - Vector: ANN with IVF_HNSW_SQ embeddings
  - Querying: RRF fusion of BM25 + vector results
  - Reranker: cross-encoder BAAI/bge-reranker-v2-m3 (568M params)

Schema index (Cond A + B):
  - First 50 rows of each structured file (CSV/JSON) only
  - Same FTS + vector pipeline as hybrid (Cond A) / BM25 only (Cond B)
  - Purpose: let agent search by data shape/column structure, not content

BM25 index (Baseline + Cond B):
  - FTS only, no vector component
  - Same 800-word chunks as hybrid index

=============================================================================
OUTPUT STRUCTURE
=============================================================================

results/{condition}/{model}/agent_results.jsonl   — per-task results
results/{condition}/{model}/eval_results.csv      — flat CSV summary
results/{condition}/{model}/tools_breakdown.csv   — per-task per-tool counts
results/traces/{condition}/{model}/{task}/        — per-call trace JSONL

=============================================================================
PRIMARY METRICS
=============================================================================

End-to-end:    EM (exact match), F1
Discovery:     D_ret (retrieval coverage), D_acc (gold dataset accessed)
               Dacc precision, Dacc recall, exploration_gap = D_ret - D_acc
Efficiency:    avg_search_calls, avg_tool_calls, cost_usd, runtime_s
               search_efficiency = EM / avg_search_calls
Failure:       grounded_success, search_failed_budget, search_failed_quality,
               search_not_read, execution_failed, hallucination
Density:       EM by reasoning density (gold doc count bins: <=2, 3-4, 5-7, 8-10, >10)
               EM by search depth (search call bins: 1, 2-3, 4-6, 7-10, 11-30)
Cond B only:   planning overhead (cycle_count vs EM, Pearson r)
