SETUP:
For our budget-constrained agentic search, we have the following constraints:
Each agent has 30 tool calls available, shared across all tool calls with the exception of: 
EXCLUDING PLANNING AND CONTEXT MANAGEMENT TOOLS AND SUBMIT_ANSWER.
It has a context limit of 200k tokens.

We have two conditions and a baseline:
0. Baseline: Basic-context, Basic-search-tool
SEARCH: search_value (BM25), search_schema (BM25 over schema index), search_prefix
CONTEXT: Prompt with tools description only (no planning tool, no skills, no summarize_context)
NOTE: Identical search tools to Condition B — isolates the effect of Condition B's context tools.

1. Condition A: Basic-context, Search-Rich
SEARCH: search_value (hybrid RRF: BM25 + vector), search_schema (hybrid over schema/first-50-rows index), search_reranked (hybrid + cross-encoder reranker), search_prefix
CONTEXT: Prompt with tools description

2. Condition B: Context-Rich, Search-Basic
SEARCH: search_value (BM25 always), search_schema (BM25 over schema/first-50-rows index), search_prefix
CONTEXT: Prompt with tools description, Planning tool, Skills plugin, Loop nudge, Context summarization tool
  - Planning tool: agent writes its own research plan as text; injected into system prompt under "## CURRENT PLAN" and persists across turns. Agent calls it again to update. Does NOT spawn a subagent.
  - Skills plugin: three on-demand strategy guides loaded by the agent via skills("name") calls (excluded from 30-call budget). Each is a SKILL.md injected into context when requested:
      * plan-agent: 4-step decomposition framework (entity resolution → dataset discovery → relational mapping → execution plan), sub-task format, replanning triggers, search reformulation strategies
      * discover-data: search tool selection guide, entity resolution to government terminology (e.g. "food stamps"→"SNAP"), dataset type identification (Wikipedia vs. datagov), lexical/granularity/proxy pivot strategies
      * query-data: tool cost ladder (query_file > grep_file > read_file > download+execute_code), output truncation recovery, column name verification, query examples
  - Loop nudge: CategoryStagnationHandler detects when the agent issues N consecutive calls in the same tool category (search / inspect / execute) and injects a steering message prompting it to switch strategies.
  - Context summarization tool (summarize_context): agent manually calls this to drop N oldest messages and anchor a written summary at position 0 in history — frees context without losing key findings.

|| SEARCH_TOOL_SETUP:

- HYBRID_SEARCH_SETUP (used by Condition A: search_value, search_reranked):
PROFILING:
For every table and text file, we ingest them into 800 word chunks inserted into a LanceDB vector database.
For tables of JSON or CSVs or delimited text, we ingest them into 10k rows or items per table.
For text-files (wikipedia and others), we ingest the whole thing.
These chunks have both an FTS index and vector embeddings built on them.
For FTS, we do a simple BM25 with "en_stem" tokenizer.
For Vector Search, we employ ANN indexes with IVF_HNSW_SQ.
QUERYING: Standard hybrid RRF (reciprocal rank fusion) of BM25 + vector results.
RERANKER: Cross-encoder, BAAI/bge-reranker-v2-m3 (568M)

- SCHEMA_SEARCH_SETUP:
A SEPARATE index built from only the first 50 rows of each table (structured files only).
Purpose: allows the agent to search by data shape/structure rather than content.
Same FTS + vector pipeline as hybrid, but ingested from the schema sample (first 50 rows).
This helps when the agent knows the kind of data it needs but not the content keywords.

- BM25_SEARCH_SETUP (used by Baseline and Condition B: search_value):
FTS-only, no vector component. BM25 with "en_stem" tokenizer over the same 800-word chunks.

=============================================================================
TASK FORMAT
=============================================================================

Tasks are JSON files under tasks_mini/{k}-d-{d}/task_N.json.
Directory naming: k = number of hops (reasoning steps), d = dataset diversity
  - d=1: single dataset type (all datagov or all Wikipedia)
  - d=3: mixed types (datagov + Wikipedia)
  - d=4: multi-step cross-source (datagov → Wikipedia → datagov, etc.)

TOP-LEVEL FIELDS:
  question          — natural-language question posed to the agent
  answer            — gold answer string (plain value, e.g. "20")
  practical_motivation — human-readable motivation for the task
  nodes             — dict of evidence nodes (see below)
  reasoning_chain   — list of strings: hop-by-hop derivation of the answer
  verdict           — "pass" if task is solvable, else "fail"
  datasets_used     — list of gold dataset IDs the agent must access to answer

NODE FORMAT (nodes["1"], nodes["2"], ...):
  source     — dataset path: "datagov/<dataset-id>/files/<file>" or
               "wikipedia/<Article_Name>/content.txt"
  fact       — the exact fact extracted from this source
  subquestion — the sub-question this node answers (may reference prior nodes)
  answer     — the answer to this subquestion (string or dict of values)

REASONING CHAIN FORMAT:
  Each entry is a string describing one hop:
    "HOP N (d=<depth>, <source type> - nodes M-P):"
    "  Node M: <fact>"
    "  Aggregation: <computation>" (when multiple nodes feed one hop)
    "  Result: <intermediate answer>"
  Final entry gives the answer.

EXAMPLE (k=3, d=4 — 3 hops, 7 nodes, 7 gold datasets):
  Question: Which DC ward had lowest avg crime 2018-2021? Find the Maryland county
            containing its neighborhoods. Report that county's avg household income
            rank among U.S. counties in 2020.
  Answer:   "20"
  Nodes 1-4: datagov crime files (2018-2021) — aggregated to find Ward 3
  Node 5:   wikipedia Neighborhoods_in_Washington — maps Ward 3 to Chevy Chase / Friendship Heights
  Node 6:   wikipedia Chevy_Chase,_Maryland — places Chevy Chase in Montgomery County, MD
  Node 7:   wikipedia Montgomery_County,_Maryland — income rank = 20th
  datasets_used: ["crime-incidents-in-2018", ..., "Chevy_Chase,_Maryland",
                  "Neighborhoods_in_Washington,_D.C.", "Montgomery_County,_Maryland"]

NOTE: datasets_used IDs are bare dataset identifiers (no prefix path). They map
to the gold set used for D_ret and D_acc metric computation.


=============================================================================
EVALUATION METRICS (DEFINITIONS)
=============================================================================

END-TO-END ACCURACY:
  EM  — Exact Match: 1 if agent answer string == gold answer string (normalized), else 0
  F1  — Token-level F1 between agent answer and gold answer tokens

DISCOVERY:
  D_ret (retrieval coverage)
      = fraction of gold dataset IDs that appeared in ANY search result during the run
        i.e. len(retrieved_gold) / len(gold_ids)
        "Did the search surface the right datasets?"

  D_acc (gold dataset accessed)
      = fraction of gold dataset IDs that the agent actually opened (peek/read/query/grep)
        i.e. len(read_gold) / len(gold_ids)
        "Did the agent actually use what it found?"

  Dacc precision  = len(read_gold) / len(all_datasets_read)   — of all datasets opened, how many were relevant?
  Dacc recall     = D_acc (same as above)                     — fraction of gold datasets opened

  exploration_gap = D_ret - D_acc
      Measures how much the agent retrieved but failed to inspect.
      High gap → agent found the right datasets but ignored them.
      Low gap  → agent inspects what it retrieves (or never found them at all).

EFFICIENCY:
  avg_search_calls   — mean number of search tool calls per task
  avg_tool_calls     — mean total tool calls per task (budget = 30, excl. plan/skills/summarize_context/submit_answer)
  cost_usd           — total USD cost (input + output tokens × per-token rate)
  runtime_s          — wall-clock seconds per task
  search_efficiency  = EM / avg_search_calls  (information value per search call; higher = better)

FAILURE TAXONOMY (mutually exclusive labels per task):
  grounded_success       — correct answer, found in data
  search_failed_budget   — wrong/no answer AND tool_calls_total >= 30 (ran out of budget)
  search_failed_quality  — wrong/no answer AND tool_calls_total < 30 (retrieval quality issue)
  search_not_read        — gold datasets retrieved but never opened by the agent
  execution_failed       — agent found and opened data but code/query error prevented extraction
  hallucination          — agent gave answer without accessing supporting data

DENSITY ANALYSIS:
  EM by reasoning density  — mean EM binned by len(datasets_used): <=2, 3-4, 5-7, 8-10, >10
  EM by search depth       — mean EM binned by search call count:   1, 2-3, 4-6, 7-10, 11-30

CONDITION B ONLY:
  planning overhead  — Pearson r(cycle_count, EM): does more planning correlate with better answers?
                       Also: mean EM and mean output_tokens per cycle_count bin [1-3, 4-6, 7-10, 11+]
